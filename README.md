给一人公司创业者的产品判断、相似案例、定价获客与交付路径。

别再问“这个点子能不能做”。先问：**谁现在付钱？为什么现在付？你一个人能不能低成本交付？**

这个 Skill 不负责给你创业鸡汤。它负责把一个想法压回商业现实：收费机制、替代方案、获客渠道、交付边界、毛利、风险和停损线。

> v0.1 public alpha：已经可以公开试用。知识库、案例库和回答样本会持续扩充，目标是做成一人公司创业者真正愿意长期使用的商业作战库。

## 三个入口

| 入口 | 适合问什么 | 输出重点 |
|---|---|---|
| `/产品判断` | 我这个产品想法能不能做？ | 付费意愿、替代方案、获客难度、交付成本、停损线 |
| `/相似案例` | 市面上有没有类似产品或路径？ | 相似案例、可复制部分、不可复制风险、下一步验证 |
| `/定价获客` | 怎么收费、怎么拿第一批用户？ | 入门价、核心价、高价版、第一单支付路径、首批渠道 |

商业化可行性是第一准则。功能、技术、品牌、愿景都要服从一个问题：**这个东西能不能收钱，并且不把一个人拖进无限交付和售后。**

## 30 秒看懂输出差异

同一个问题：

```text
我想做一个 Markdown 转 HTML 的产品，能不能做？
```

普通大模型很容易回答：

```text
可以先做 MVP，找到目标用户，持续优化体验，打造差异化。
```

一人公司Skill 应该追问和判断：

```text
先别把“格式转换”当成生意。市面上已经有免费转换器、编辑器内置导出、文档站生成器、
Pandoc/markdown-it/remark 这类开源工具、CloudConvert/ConvertAPI 这类转换 API、邮件编辑器和开发者工具链。

真正可能付费的不是“转 HTML”，而是：
1. 技术文档/产品说明能稳定发布到官网、邮件或帮助中心；
2. 非技术团队不用碰代码也能保持样式一致；
3. 内容发布节省人工排版和反复修改。

建议先卖一个更窄的结果：
“把 10 篇 Markdown 文档变成可发布的品牌化帮助中心页面”，收 199/499 元服务费。
如果 20 个目标用户里没有 3 个愿意为这个结果付钱，就不要先做 SaaS。
```

差别不在语气更狠，而在信息增量：它必须说清楚竞品、替代工作流、付费触发、第一单、交付边界和停损线。

## 为什么不是提示词合集

提示词只能改变表达方式，不能自动带来商业判断。

这个仓库真正要沉淀的是三层东西：

- **判断框架**：先判断收费、触达、交付和停损，而不是先讨论功能。
- **案例情报**：用真实案例、替代方案、GitHub 实操信号和内容平台观察校准判断。
- **执行现实**：把备案、支付、微信生态、平台规则、发票、私域和售后成本放进同一个判断里。

所以它的目标不是“让 AI 回答得更像商业顾问”，而是让回答背后有可复用的商业证据和行动路径。

## 快速开始

把 `skills/one-person-company` 复制到你的 Agent / Codex / ChatGPT-like skills 目录，然后这样提问：

```text
/产品判断
idea: 我想做一个 markdown 转 html 的产品
target_user: 写技术文档、博客、产品说明的人
workflow: 现在需要把 markdown 转成可发布的网页或邮件格式
paid_trigger: 用户为什么现在愿意付钱还不确定
```

```text
/相似案例
idea: 我想做一个面向小红书商家的 AI 选题工具
target_user: 小红书商家、代运营、个人博主
use_case: 每周找选题、写标题、判断爆款角度
market_hint: 不知道
```

```text
/定价获客
idea: 我想做一个一人公司案例检索库
target_user: 独立开发者、自由职业者、小团队创业者
paid_trigger: 做产品前想减少误判
acquisition_channel: GitHub、公众号、即刻、微信群
```

## 案例情报库与知识库规模

这个项目的核心资产不是提示词，而是案例情报库。

当前内置：

- 100+ 标准化案例
- 30 条 gold cases
- 39 个案例来源
- 10 个 GitHub 高价值开源知识源
- 16 条 GitHub 实操信号
- 10 个高频市场模式
- 13 个回答质量评估场景
- 13 条金标回答样本

当前已覆盖的高频市场模式包括：转换工具/API、AI 电商素材/虚拟试衣、AI 小红书内容、开发者工具/API、知识产品、案例情报库、AI 自动化服务、Notion 建站、本地获客、模板与 boilerplate。

案例不是用来装饰回答的。每条资料都要拆成：

- 目标用户
- 付费机制
- 获客路径
- 交付方式
- 可复制部分
- 不可复制风险
- 本土执行现实

它默认工作在中文商业语境里：微信生态、小红书、公众号、知乎、即刻、私域、备案、支付、发票、平台规则、内容分发和信任成本都可能影响判断。海外案例可以作为商业机制参照，但不能直接照搬。

## 它适合谁

- 有产品想法，但不想被“先做 MVP”这种空话浪费时间的人
- 正在做独立开发、自媒体、咨询、模板、社群、数据产品或 AI 小工具的人
- 想用一个人能承受的方式验证收费、交付和获客的人
- 需要案例、竞品和替代方案来辅助商业判断的人

它不适合：

- 只想听鼓励、不想面对付费现实的人
- 想把一个普通功能包装成大平台的人
- 没有目标用户、没有渠道、也不愿意做真实验证的人

## 本地工具

安装依赖：

```bash
npm install
```

校验仓库：

```bash
npm test
```

匹配相似案例：

```bash
node scripts/opc/match-product-idea.js "我想做一个 AI 小红书选题助手"
```

校验案例库：

```bash
npm run opc:validate:cases:seed
```

校验 GitHub 实操信号：

```bash
npm run opc:validate:github-sources
```

## 质量门槛

好的回答应该：

- 简洁，但不是空泛。
- 直接，但不是情绪化。
- 可以反驳用户，并给出商业理由。
- 用案例、竞品、替代方案或渠道事实提供信息增量。
- 说清楚备案、支付、平台分发、微信生态、交付信任和合规摩擦。
- 拆清楚商业模式：服务、模板、工具、咨询、社群、数据、自动化，不能混在一起讲。
- 说明第一单如何支付、交付边界是什么、毛利和售后风险在哪里。
- 明确今天能做的验证动作，以及停损线。

坏回答通常长这样：

- 建议用户先做 MVP，但没有说卖给谁。
- 建议持续输出，但没有渠道、内容角度和成交路径。
- 说市场很大，但没有现有付费机制。
- 只分析功能，不分析获客和交付成本。
