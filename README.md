大多数创业建议死在一句话里：先做 MVP。

这个 Skill 从不这样回答。

它不负责鼓励你创业。它负责把一个想法压回商业现实：谁现在付钱，他现在用什么替代方案，竞品怎么收费，你一个人能不能交付，第一单怎么收，什么时候该停。

> v0.1 public alpha：已经可以公开试用。知识库、案例库和回答样本会持续扩充，目标是做成一人公司创业者真正愿意长期使用的商业作战库。

## 它判断什么

一人公司Skill 是一个专门判断“小产品能不能变成生意”的商业作战库。

它默认把每个想法拆成七件事：

- 直接竞品：已经有人在卖什么。
- 相邻替代：用户现在怎么绕过去。
- 免费替代：用户为什么可以不付钱。
- 高价替代：预算已经流向哪里。
- 收费机制：钱因为什么动作发生。
- 交付边界：一个人能不能低成本兑现承诺。
- 停损线：什么信号说明应该暂停、收窄或放弃。

商业化可行性是第一准则。功能、技术、品牌、愿景都要服从一个问题：**这个东西能不能收钱，并且不把一个人拖进无限交付和售后。**

## 三个入口

| 入口 | 适合问什么 | 输出重点 |
|---|---|---|
| `/产品判断` | 我这个产品想法能不能做？ | 付费意愿、替代方案、获客难度、交付成本、停损线 |
| `/相似案例` | 市面上有没有类似产品或路径？ | 相似案例、可复制部分、不可复制风险、下一步验证 |
| `/定价获客` | 怎么收费、怎么拿第一批用户？ | 入门价、核心价、高价版、第一单支付路径、首批渠道 |

## 三个例子

同样是一个产品想法，普通 AI 往往会给你：

```text
这个方向不错，可以先验证需求，再优化体验和差异化。
```

一人公司Skill 不这样回答。

### 1. AI 换装小程序

```text
/产品判断 我想做一个换装小程序，用户输入一张人物图和一张商品图，就可以换衣服。
```

它会先判断：不要先做泛 C 端娱乐换装。真正有可能付费的是商家商品图、买家秀替代、详情页素材、小红书种草图和广告测试图。

它会追问：API 成本能不能被 99/199/499 元服务包覆盖？用户是淘宝/小红书/私域商家，还是只想免费玩一下的普通用户？生成失败、衣服变形、肖像和平台审核风险谁承担？

第一步不是做完整小程序，而是手工服务 10 个卖家：每人 5 张商品图，交付 20 张可发布素材。10 个卖家里没有 3 个愿意付 99 元，就不要继续烧 API。

### 2. Markdown 转 HTML

```text
/产品判断 我想做一个 Markdown 转 HTML 的产品。
```

它不会把“能转格式”当成生意。它会先拆替代方案：Marked 2、Markdown Monster、CloudConvert、ConvertAPI、Pandoc、markdown-it、remark、VS Code 插件、文档站、CMS、邮件编辑器、帮助中心工具。

真正可能付费的不是“转 HTML”，而是稳定发布、批量处理、样式一致、API 自动化、隐私或本地处理、团队协作。

可验证切口是：把 10 篇 Markdown 文档变成可发布的品牌化帮助中心页面，收 199/499 元服务费。20 个目标用户里没有 3 个愿意给真实文件并付款，就不要先做 SaaS。

### 3. 一人公司案例库

```text
/定价获客 我想做一个一人公司案例库。
```

它会先反驳：案例本身不值钱，减少错误决策才值钱。

可收费的不是“我收集了很多案例”，而是“我能用相似案例告诉你这个想法哪里会死，第一单怎么收，什么情况下该停”。所以更好的第一单不是订阅制知识库，而是产品判断报告、相似案例诊断、定价获客方案或小额陪跑。

停损线也要清楚：如果用户只收藏案例、不愿为具体判断付费，说明信息焦虑存在，但商业支付点不成立。

## 为什么不是提示词合集

提示词只能改变表达方式，不能自动带来商业判断。

这个仓库真正沉淀的是三层东西：

- **判断协议**：先判断收费、触达、交付和停损，而不是先讨论功能。
- **案例情报**：用真实案例、替代方案、GitHub 实操信号和内容平台观察校准判断。
- **执行现实**：把备案、支付、微信生态、平台规则、发票、私域、内容分发和售后成本放进同一个判断里。

所以它的目标不是“让 AI 回答得更像商业顾问”，而是让回答背后有可复用的商业证据和行动路径。

## 证据资产

这个项目的核心资产不是提示词，而是案例情报库。

当前内置：

- 100+ 标准化案例
- 30 条 gold cases
- 39 个案例来源
- 10 个 GitHub 高价值开源知识源
- 16 条 GitHub 实操信号
- 10 个高频市场模式
- 13 个回答质量评估场景
- 13 条金标回答样本

当前已覆盖的高频市场模式包括：转换工具/API、AI 电商素材/虚拟试衣、AI 小红书内容、开发者工具/API、知识产品、案例情报库、AI 自动化服务、Notion 建站、本地获客、模板与 boilerplate。

案例不是用来装饰回答的。每条资料都要拆成：

- 目标用户
- 付费机制
- 获客路径
- 交付方式
- 可复制部分
- 不可复制风险
- 本土执行现实

## 快速开始

把 `skills/one-person-company` 复制到你的 Agent / Codex / ChatGPT-like skills 目录，然后这样提问：

```text
/产品判断
idea: 我想做一个 AI 小红书选题工具
target_user: 小红书商家、代运营、个人博主
workflow: 每周找选题、写标题、判断爆款角度
paid_trigger: 不确定用户为什么现在愿意付钱
```

```text
/相似案例
idea: 我想做一个一人公司案例检索库
target_user: 独立开发者、自由职业者、小团队创业者
use_case: 做产品前想减少误判
market_hint: 不知道案例库能不能收费
```

```text
/定价获客
idea: 我想做一个给本地商家的 AI 获客工具
target_user: 本地门店、服务商家、小团队
paid_trigger: 想知道第一单怎么收
acquisition_channel: 微信群、私域、小红书、线下转介绍
```

## 本地工具

安装依赖：

```bash
npm install
```

校验仓库：

```bash
npm test
```

匹配相似案例：

```bash
node scripts/opc/match-product-idea.js "我想做一个 AI 小红书选题助手"
```

校验案例库：

```bash
npm run opc:validate:cases:seed
```

校验 GitHub 实操信号：

```bash
npm run opc:validate:github-sources
```

## 质量门槛

好的回答应该：

- 简洁，但不是空泛。
- 直接，但不是情绪化。
- 可以反驳用户，并给出商业理由。
- 用案例、竞品、替代方案或渠道事实提供信息增量。
- 说明第一单如何支付、交付边界是什么、毛利和售后风险在哪里。
- 拆清楚商业模式：服务、模板、工具、咨询、社群、数据、自动化，不能混在一起讲。
- 明确今天能做的验证动作，以及停损线。

坏回答通常长这样：

- 建议用户先做 MVP，但没有说卖给谁。
- 建议持续输出，但没有渠道、内容角度和成交路径。
- 说市场很大，但没有现有付费机制。
- 只分析功能，不分析获客和交付成本。
