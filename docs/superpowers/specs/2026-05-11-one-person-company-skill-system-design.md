## 0. 命名与一句话定位

### 0.1 推荐名称

**一人公司内参**

推荐理由：

- 比“作战系统”更像一个高信任商业产品，不像普通工具站。
- “内参”传达的是内部判断、真实坑位、案例底色和少数人信息差，符合本项目的稀缺性。
- 名称足够短，适合做官网、技能名、内容栏目和付费产品。

备选名称：

| 名称 | 优点 | 问题 |
|---|---|---|
| 一人公司内参 | 信任感强，专业，适合长期品牌 | 需要内容质量支撑，否则容易显得空 |
| 一人公司参谋部 | 有陪伴和判断感，适合工具箱 | 稍微偏组织化 |
| 独立生意内参 | 更商业、更宽 | “一人公司”定位弱一点 |
| 单干商业参谋 | 更锋利，更接地气 | 品牌质感略粗 |
| 一人公司真相库 | 差异化强 | 容易被理解成纯避坑资料库 |

当前建议：产品对外暂定 **一人公司内参**；内部 skill 代码仍使用 `opc`，因为短、稳定、方便扩展。

### 0.2 一句话定位

**一人公司内参** 是一个面向国内一人公司创业者的专业商业判断 skill 工具箱。

它不做泛泛的创业陪聊，不靠语气制造“高质量感”，而是用结构化知识库、案例拆解、现实坑位库和阶段路线，帮助用户判断：

- 这个生意能不能做
- 当前最核心的商业断点在哪里
- 内容、自媒体、产品、交付、支付、合规、渠道之间怎么取舍
- 国内真实环境下有哪些坑，怎么权衡
- 下一阶段应该怎么规划，而不是收到一堆漂亮但不会执行的建议

## 1. 为什么要做这个

原 Book of Elon 的问题不是 UI 或付费墙，而是价值对象太泛：

- “Elon 风格 AI 教练”容易被任意大模型 prompt 复刻
- 对话质量依赖模型临场发挥，缺少知识结构
- 网站公开开放会带来 API 成本和风控问题
- 用户需求没有被压到一个具体、强痛感、愿意付费的场景

新方向的机会在于：

- 国内一人公司创业者正在增加，但他们面对的是非常具体的国内约束：备案、支付、平台规则、私域、内容封控、主体资质、税务、服务器、交付边界、AI 成本、获客成本。
- 普通大模型会给出“去备案、做 SEO、找用户调研、持续输出、打造个人品牌”这类正确但空泛的建议。
- 真正有价值的回答必须结合第一性原理、商业模型、国内现实、案例经验和执行阻力。

这个产品要卖的不是“建议”，而是 **判断质量**。

## 2. 第一性原理

### 2.1 一人公司的根本矛盾

一人公司的根本矛盾是：

> 一个人的时间、精力、技能、资金、心理承受力都有限，但一个商业闭环需要需求、获客、转化、交付、复购、客服、合规、财务、技术、内容全部成立。

所以一人公司不是“小公司”，而是一个特殊结构：

- 必须降低协作成本
- 必须压低固定成本
- 必须避免重交付
- 必须用自动化、内容、软件、模板、SOP 或平台能力放大自己
- 必须减少维护债和客服债
- 必须尽早证明能收钱

### 2.2 核心判断公式

一人公司能不能成立，不看“想法好不好”，看这条等式：

```text
一人公司成立概率 =
  可触达需求
× 付费意愿
× 毛利
× 信任
× 复购/持续性
× 个人交付能力
÷ 获客成本
÷ 维护复杂度
÷ 平台/合规风险
÷ 创始人执行阻力
```

每次回答都要判断：用户的问题到底卡在这个公式的哪一项。

### 2.3 三个优先级

1. **商业闭环优先于表达好听**
   回答必须能落到人、货、场、价格、交付、渠道、成本、风险。

2. **真实约束优先于理想路径**
   如果国内备案、微信支付、平台风控、私域封号、个人主体限制会影响路径，必须讲清楚权衡。

3. **可执行阻力优先于正确建议**
   “找 10 个用户访谈”可能是正确建议，但多数人不会做。skill 要给低阻力版本，同时说明代价。

## 3. 产品形态

参考 `dbskill` 的启发，本产品不应是单一 prompt，而应是多 skill 工具箱 + 知识库。

### 3.1 v0.1 最小工具箱

第一版只做 5 个核心 skill，不一次性铺太大。

| Skill | 作用 | 触发 |
|---|---|---|
| `opc` | 主入口，识别问题并路由 | “帮我看看这个一人公司项目” |
| `opc-diagnosis` | 商业诊断，判断断点 | “我这个项目能不能做/为什么没人买” |
| `opc-content` | 自媒体与内容商业诊断 | “小红书/抖音/公众号怎么做内容变现” |
| `opc-benchmark` | 案例和对标拆解 | “我该学谁/这个模式有没有对标” |
| `opc-china-reality` | 国内现实坑位与权衡 | “备案/支付/服务器/平台规则怎么选” |

| `opc-report` | 生成诊断报告 |

## 4. 知识库结构

### 4.1 这套结构到底是什么意思

你不需要把 `atoms.jsonl + packs/*.md` 理解成技术方案，它本质上是在解决一个产品问题：

> 怎么让回答长期稳定地高质量，而不是每次靠大模型临场发挥。

简单说：

- `packs/*.md` 是“方法论手册”。它放稳定框架，比如一人公司公式、内容商业判断、国内坑位清单、案例拆解模板。模型回答前先看方法论，不会乱飘。
- `atoms.jsonl` 是“判断卡片库”。每一行是一条经过提炼的事实、坑位、案例结论或市场观察，并带来源、适用阶段、风险和可信度。
- `SKILL.md` 是“使用说明书”。它告诉模型什么时候该用哪个框架、怎么组织答案、哪些空话不能说。

为什么不只写一个长 prompt：

- 长 prompt 会越来越乱，后期很难维护。
- 国内政策、平台规则、支付和备案会变化，需要单条更新，而不是重写整套 prompt。
- 案例和坑位会越来越多，必须能按主题、阶段、路线检索。
- 想做到“专业”，就必须把知识沉淀成资产，而不是把希望押在模型聪明上。

一个具体例子：

用户问：“我做官网是不是要先备案？”

普通模型可能回答：“如果面向中国大陆用户，建议进行 ICP 备案。”

本产品应该先从 `packs/china-realities.md` 读取备案权衡框架，再从 `atoms.jsonl` 找到“香港服务器无需 ICP 备案”“大陆服务器备案与主体、接入商有关”“验证期不宜过早承担备案摩擦”等判断卡片，最后给出：

```text
如果你还没证明有人愿意付费，备案不是第一优先级。
验证期先用香港/新加坡节点或第三方承接页，成本更低、摩擦更小。
但如果你要长期做品牌官网、接入大陆云资源、跑广告或小程序生态，迟早要处理主体和备案。
停损条件：连续 2 周没有有效咨询或付款信号，不要把时间花在备案和官网完善上。
```

这就是知识库的价值：不是让答案更长，而是让答案更准、更稳、更像懂国内实战的人。

### 4.3 知识原子格式

模仿 dbskill 的原子库思想，但字段服务于“一人公司”。

```json
{
  "id": "opc_2026_0001",
  "knowledge": "验证期不应默认选择大陆备案；若主要目标是快速验证需求，香港或新加坡节点通常更低摩擦。",
  "original": "阿里云/腾讯云官方文档均说明中国香港及境外服务器无需 ICP 备案。",
  "source": "https://cloud.tencent.cn/document/product/243/18908",
  "source_type": "official_doc",
  "date_checked": "2026-05-11",
  "topics": ["china-reality", "hosting", "icp"],
  "routes": ["software", "content", "service"],
  "stage": ["validation", "launch"],
  "risk": ["policy", "platform", "ops"],
  "type": "constraint",
  "confidence": "high"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `knowledge` | 提炼后的可复用判断 |
| `original` | 原始来源摘要，不长篇复制 |
| `source` | 官方文档、案例文章、访谈、项目链接 |
| `source_type` | official_doc / case / interview / book / founder_note / market_observation |
| `topics` | 内容、支付、备案、平台、定价、交付等 |
| `routes` | 软件工具、知识产品、自动化服务、自媒体、私域等 |
| `stage` | idea / validation / first-revenue / acquisition / delivery / scale |
| `risk` | policy / platform / cost / legal / execution / delivery |
| `confidence` | high / medium / low |

### 4.4 可信度规则

- 官方规则类内容必须有官方来源或注明需复查。
- 案例类内容必须区分“公开事实”和“推断”。
- 个人经验类内容必须标为 `market_observation`，不能包装成确定规律。
- 法务、财税、合规只能做方向提醒，不能替代专业意见。
- 所有容易随时间变化的内容必须有 `date_checked`。

## 5. 核心知识域

### 5.1 商业底层模型

必须覆盖：

- 需求：用户是否已经为问题付出代价
- 渠道：是否能低成本触达买家
- 信任：凭什么相信你
- 定价：价格本身如何筛选用户和定义产品
- 毛利：一人公司必须避免低毛利高客服
- 交付：是否能被 SOP、模板、软件、AI、外包放大
- 复购：是否有持续购买或持续使用理由
- 现金流：何时收钱，何时付成本
- 维护债：软件、社群、课程、服务都会产生维护债
- 创始人状态：不是心理安慰，而是执行瓶颈和能力边界

### 5.2 自媒体与内容商业

这是产品重点，不是附属模块。

要区分：

- 内容为了涨粉
- 内容为了建立信任
- 内容为了筛选客户
- 内容为了成交
- 内容为了承接私域
- 内容为了复购和转介绍

必须避免空话：

- “持续输出”不算建议
- “打造个人品牌”不算建议
- “做小红书/抖音”不算建议
- “发干货”不算建议

回答要落到：

- 内容服务哪个产品
- 目标用户为什么会停下来看
- 内容承诺了什么结果
- 评论/私信/付款如何承接
- 平台规则如何影响形式
- 是否需要露脸、直播、投流、矩阵、私域

### 5.3 国内现实坑位

第一批必须覆盖：

| 领域 | 坑位 |
|---|---|
| 备案与服务器 | 大陆备案、香港服务器、新加坡节点、CDN 回源、域名实名、接入商限制 |
| 支付 | 微信/支付宝官方商户、个人收款、聚合支付、虎皮椒/易支付风险、退款、对账、封控 |
| 公司主体 | 个体户、有限公司、个人收款、开票、税务、经营范围 |
| 微信生态 | 公众号、小程序、个人号、企微、微信群、风控、诱导分享、外链限制 |
| 小红书 | 内容种草、私信限制、引流风险、薯条、聚光、矩阵、账号权重 |
| 抖音 | 短视频、直播、千川、商品/服务转化、强平台依赖 |
| 私域 | 加微信路径、社群交付、朋友圈成交、封号风险、客服压力 |
| AI 工具 | API 成本、模型降级、提示词商品化、工具同质化、交付为结果而非 AI |
| 内容合规 | 医疗、金融、教育、玄学、赚钱类内容、案例夸大、虚假承诺 |
| 交付 | 咨询、训练营、课程、模板、软件、代运营、自动化服务的交付债 |

回答中遇到这些主题，必须讲 **权衡**，而不是单一建议。

示例：

```text
大陆服务器：访问和信任更好，但备案、主体和内容风险更重。
香港服务器：验证更快，但访问速度、微信环境和长期信任会打折。
如果你还没证明有人付钱，先香港/新加坡验证；如果已有稳定成交，再考虑大陆备案。
```

### 5.4 案例库

案例不用于励志，只用于拆商业结构。

第一批案例类型：

- 国外一人公司 / indie hacker：Pieter Levels、Justin Welsh、Daniel Vassallo、Arvid Kahl、Marc Lou 等
- 国内一人公司方法论与实践：EasyChen、一人公司方法论、独立开发者、知识付费、私域、内容 IP、AI 工具案例
- dbskill / dontbesilent 作为结构参考：原子库、多 skill 路由、诊断与内容工具箱
- 失败案例：做完产品没人买、内容涨粉不变现、低价课交付拖死、工具订阅无人续费、私域被封、备案/支付卡住

## 6. 阶段路线

### 6.1 阶段定义

| 阶段 | 核心问题 | 不该做什么 |
|---|---|---|
| 方向期 | 哪类人、哪类痛、哪类钱 | 不要先做完整产品 |
| 验证期 | 是否有人愿意停下/咨询/付钱 | 不要投入重开发、备案、复杂支付 |
| 第一笔收入 | 如何收第一笔钱 | 不要追求规模化和自动化 |
| 可重复获客 | 哪个渠道能稳定带来买家 | 不要同时做多个平台 |
| 交付系统化 | 如何降低交付和客服成本 | 不要继续堆人工服务 |
| 利润化 | 如何提升毛利和复购 | 不要只看收入不看精力 |
| 风险治理 | 如何处理合规、主体、税务、平台风险 | 不要把临时方案当长期资产 |

### 6.2 路线类型

| 路线 | 典型产品 | 最大优势 | 最大风险 |
|---|---|---|---|
| 软件工具 | SaaS、小程序、插件、网页工具 | 交付轻、边际成本低 | 获客难、同质化、维护债 |
| AI 服务 | 自动化方案、智能体、代交付 | 容易先手工交付验证 | 容易变外包，交付重 |
| 知识产品 | 课程、手册、训练营、社群 | 毛利高、内容可获客 | 信任难、交付和退款风险 |
| 咨询产品化 | 诊断、陪跑、顾问包 | 高客单、快收钱 | 时间不可规模化 |
| 内容 IP | 自媒体、 newsletter、直播 | 信任和渠道可复用 | 周期长，平台依赖强 |
| 私域卖货 | 选品、团购、资料、服务 | 现金流快 | 供应链、售后、封控 |
| 本地轻服务 | 本地商家服务、获客、代运营 | 需求具体、可面谈 | 人工重，难规模化 |

## 7. 回答协议

### 7.1 默认流程

每次回答按这个顺序组织，但不机械显示所有标题：

1. **判阶段**
   用户处在方向期、验证期、第一笔收入、获客期、交付期、利润化、风险治理哪一段。

2. **判断点**
   对应核心公式中哪一项断了：需求、渠道、信任、定价、毛利、交付、复购、风险、执行阻力。

3. **讲商业逻辑**
   用第一性原理解释为什么这里是主矛盾。

4. **调案例或现实坑位**
   至少引用一个案例结构、国内现实约束或可验证来源；没有把握就说不确定。

5. **给方案权衡**
   不只给 A，要说明 A/B 的代价。

6. **给下一步**
   给一个低阻力但能产生信号的动作。若用户需要规划，给 1/7/30 天路线。

7. **明确停损条件**
   告诉用户什么信号出现就该暂停、换方向或降级方案。

### 7.2 输出风格

- 简洁，有力，但不装狠
- 先判断，再解释
- 少安慰，多约束
- 少抽象，多权衡
- 能用表格时用表格
- 不给 10 条建议；最多 1 个主判断 + 2 个可选路径 + 1 个下一步

### 7.3 禁止句式

除非马上转成具体动作，否则禁止：

- 你需要做用户调研
- 打造个人品牌
- 持续输出
- 找到痛点
- 做 MVP
- 做私域
- 先做 SEO
- 去备案
- 接入支付
- 多发小红书
- 做差异化
- 坚持下去

这些词不是不能用，而是不能作为结论。

## 8. 质量评测 Rubric

每个回答 10 分。

| 维度 | 分值 | 标准 |
|---|---:|---|
| 商业本质 | 2 | 是否指出需求、渠道、定价、毛利、交付、复购等核心变量 |
| 国内现实 | 2 | 是否讲到平台、支付、备案、主体、私域、政策等真实约束 |
| 案例底色 | 1.5 | 是否用案例结构或来源支撑，而不是空口判断 |
| 路线规划 | 1.5 | 是否给出阶段判断和路径，而不是单点建议 |
| 可执行性 | 1.5 | 下一步是否足够低阻力，用户真的可能做 |
| 表达质量 | 1 | 是否简洁、有力、无废话 |
| 不确定性诚实 | 0.5 | 涉及政策/平台规则时是否提示需复查 |

低于 7 分的回答不可接受。

### 8.1 直接判不合格

- 只给情绪价值，没有商业判断
- 只讲理论，没有国内现实
- 建议用户做大动作，但没有说明阻力和替代路径
- 用“持续输出/打造 IP/做 MVP/找痛点”作为结论
- 编造案例或来源
- 涉及政策、支付、备案却不提示时效性

## 10. 外部项目研究摘要

### 10.1 dbskill

可学习：

- 原子库：从大量内容中提炼结构化知识点
- 多 skill：主入口路由，不让一个 skill 承担所有任务
- 方法论包 + 案例包：让回答不是临场发挥
- 状态管理：诊断可以保存、恢复、生成报告
- 反空话：强调消解问题，而不是回答错误问题

不能照抄：

- 其语气和公理属于作者个人风格，我们需要自己的“一人公司”商业系统
- 其内容来源、许可和品牌不能直接商业化搬运
- 其强诊断风格不一定适合所有一人公司创业者，需要保留专业而非表演性攻击

### 10.2 一人公司方法论

可学习：

- 一人公司的核心矛盾是精力有限而业务链条完整
- 以租代建、分离非核心业务、选择小市场、利用平台生态
- 营销优先 vs 开发优先的真实取舍
- 自媒体渠道不是免费，消耗大量时间和认知

注意：

- 原文 CC-BY-NC-SA，商业产品中不能直接复制派生使用；可作为研究参考，需建立自己的表达和来源体系。

### 10.3 Awesome Indie Hackers / 国外资源地图

可学习：

- 案例和资源地图有价值，但不能直接变成判断
- 国外案例需要做中国迁移风险评估：支付、平台、语言、获客、文化、合规都不同

### 10.4 GitHub 上的空位

公开项目里有：

- startup playbook
- indie hacker resources
- AI startup OS
- business model canvas
- 一人公司方法论
- dbskill 这类商业诊断工具箱

但缺少：

> 面向国内一人公司创业者，把第一性原理、内容商业、国内现实坑位、案例拆解和实践路线结合起来的专业 skill 工具箱。

这就是本项目的产品空位。

## 12. 旧 Book of Elon 项目退场原则

用户已确认：原 Book of Elon 项目放弃，删除项目，只保留有用资产。

但这个动作必须极其谨慎，因为同一台服务器里还有其他上线项目。这里的原则是：

> 不直接在服务器上删除任何东西。先在本地仓库完成资产筛选和迁移设计，再做服务器级清单确认。

### 12.1 可以保留的资产

| 资产 | 保留原因 | 处理方式 |
|---|---|---|
| 登录、限额、成本控制相关代码 | 后续如果做付费产品或内测，可复用风控经验 | 作为参考，不直接沿用 |
| paywall/onboarding 方案文档 | 里面有成本失控教训 | 移入 `docs/archive/` 或提炼进新商业设计 |
| 架构文档、事故复盘、成本 spike runbook | 对新产品很有价值 | 提炼为“API 成本控制”和“产品风控”资产 |
| 前端交互经验 | 可作为未来前端体验参考 | 暂不改版，只归档 |
| prompt/对话结构 | 可提炼出少量表达经验 | 去掉 Elon 品牌依赖，只保留方法结构 |

### 12.2 应删除或退场的资产

| 资产 | 原因 | 安全处理 |
|---|---|---|
| Elon 品牌、书籍、PDF/epub 依赖 | 新产品定位不同，可能带来版权和差异化问题 | 从新产品路径中移除，归档前确认是否仍需本地备份 |
| 原公共 AI 教练入口 | 成本不可控，价值对象太泛 | 不再作为主线开发 |
| 原 lifetime / unlimited 付费设想 | 经济模型不成立 | 文档归档，不进入新方案 |
| 与新 skill 无关的 UI 文案 | 容易污染定位 | 后续改造时删除 |

### 12.3 服务器安全规则

任何服务器清理必须按这个顺序走：

1. **识别范围**：只处理 Book of Elon 对应目录、进程、域名、Nginx 配置、PM2 配置、数据库和环境变量。
2. **列出清单**：先输出将要影响的文件、目录、端口、域名、进程名和数据库文件。
3. **确认归属**：逐项确认不属于其他上线项目。
4. **备份再动**：备份代码目录、数据库、`.env`、Nginx 配置和 PM2 配置。
5. **先停用，不删除**：先下线 Book of Elon 服务，观察其他项目是否正常。
6. **延迟删除**：稳定观察后，再删除或压缩归档。

禁止动作：

- 禁止在服务器上运行全局递归删除。
- 禁止删除未确认归属的 `nginx`、`pm2`、`node_modules`、数据库、日志目录。
- 禁止凭目录名猜测项目归属。
- 禁止为了“清爽”删除看不懂的文件。

### 12.4 本地仓库改造原则

本地仓库可以逐步变成新产品仓库，但第一阶段不做大删除：

1. 先创建新 skill/knowledge 目录。
2. 把有价值文档迁移或摘录到 `docs/archive/`、`docs/superpowers/specs/`、`knowledge/`。
3. 明确哪些旧文件被新产品替代。
4. 等新 skill 最小版本通过压力测试，再删除旧代码。

这样做的原因很简单：现在最值钱的是新产品判断系统，不是立刻把旧项目清空。

## 13. 成功标准

v0.1 成功不是“能回答”，而是：

1. 对 20 个压力测试问题，平均评分 >= 8/10。
2. 每个回答都能指出商业断点，而不是泛泛建议。
3. 遇到国内现实问题，能给出权衡，而不是单一路径。
4. 回答能引用案例结构或来源，不靠空口权威。
5. 用户读完能感觉“这不是普通大模型会说的话”。
