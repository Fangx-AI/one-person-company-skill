# Launch Readiness Plan

## 当前状态

当前项目已经不是“纯原型”状态，已经具备一版可上线基础，但还没有形成完整的生产交付闭环。

### 已完成

- `server.js` 已提供：
  - `GET /health`
  - `GET /ready`
  - `GET /config.js`
  - `POST /api/chat`
- 聊天链路已具备：
  - IP 限流
  - 突发请求限制
  - 请求体校验与裁剪
  - 上游超时控制
  - 熔断
  - 本地降级回复
  - 基础 JSON 请求日志
  - 安全缓存策略
- 前端已具备：
  - 卡片浏览
  - 直接提问
  - 本地匿名续聊

### 仍然缺少

- 可重复部署说明
- Docker 化交付
- 基础冒烟脚本接入统一脚本入口
- 生产告警与日志平台
- 多实例共享限流与缓存
- 更明确的运行模式说明（必须通过 `server.js` 提供 `/config.js` 才能启用 LLM）

## 当前主要风险

### P0 风险

1. 文档与实现脱节

- 旧版本清单已经不再反映真实代码状态
- 会误导上线判断和排期

2. 生产交付闭环不完整

- 缺少 Docker / 部署手册 / 统一脚本
- 交接给别人上线时容易走错

3. 单实例内存状态的结构性边界

- 当前限流、缓存、熔断状态都在内存里
- 单实例起步没问题
- 多实例放量时会失去全局一致性

4. 监控告警不足

- 虽然已经有结构化日志
- 但还没有对接外部平台和告警规则

### P1 风险

1. 运行模式容易误解

- 如果不是通过 `server.js` 同源提供页面
- 前端可能拿不到 `/config.js`
- 会退回本地知识模式

2. 成本控制还不够成熟

- 目前已有限流和缓存
- 但还没有跨实例缓存、配额层和更精细的流量分层

3. 用户与业务数据缺失

- 还没有真正的数据留存、指标看板和行为分析

## 上线前优先级

### 第一层：现在立刻做

1. 补齐交付基础文件

- `README.md`
- `Dockerfile`
- `.dockerignore`
- 统一 npm scripts

2. 修正文档

- 明确当前真实能力
- 明确仍未完成的事项
- 避免继续按旧假设决策

3. 保证本地与部署可验证

- 本地启动命令清晰
- 冒烟测试有固定入口
- 健康检查路径明确

### 第二层：上线首版要做

1. 对接日志平台与告警

- 关注 5xx
- 关注熔断开启
- 关注上游超时
- 关注降级比例

2. 补部署运行规范

- HTTPS
- 反向代理
- 环境变量注入
- 进程托管

3. 补产品层必需缺口

- 把 `starter_questions` 接入前端
- 补主题库更完整入口
- 明确当前是 `DeepSeek` 对话还是本地知识模式

### 第三层：准备放量时做

1. Redis

- 共享限流
- 共享缓存
- 为多实例做准备

2. 指标与分析

- 热门问题
- 热门卡片
- 模型成本
- 降级比例

3. 用户体系与配额

- 先做匿名配额
- 后续再决定是否引入登录

## 现在最适合直接开工的工程任务

把项目补到“可重复部署、可验证、可交接”的状态。

具体顺序：

1. 统一脚本入口
2. 增加 Docker 交付
3. 写清本地启动和部署说明
4. 用冒烟脚本验证部署后聊天链路

## 一句话结论

当前项目下一阶段不该继续沉迷小的 UI 细节，而应该先把“能稳定交付上线”这件事补完整。
